## Problem
서비스의 핵심 여정(로그인 → 내 문서 업로드 → 챗으로 문서에 질문) 3개 화면이 아직 화면으로 존재하지 않는다. 백엔드 연동 전, 실제 사용자·이해관계자·팀 내부가 "우리 서비스가 어떻게 생겼고 어떻게 흐르는지"를 합의할 수 있는 공통 기준이 없다. 디자인 시스템(중립 팔레트·해요체·테두리 중심의 절제된 UI)은 정의되어 있지만 아직 제품에 적용된 적이 없다.

## Goals
- 사용자가 로그인 → 내 문서 → 챗 세 화면을 끊김 없이 이동하며 서비스 전체 윤곽을 체감할 수 있다.
- 디자인 시스템의 중립적·절제된 톤(테두리 중심, 그림자 최소, 한국어 해요체)이 모든 화면에 일관되게 드러난다.
- 이후 API 연동 작업이 "이 화면의 이 자리에 이 데이터가 들어온다" 수준으로 명확해진다.
- 로그인/비로그인 영역이 시각적으로 구분되어, 인증 이후에만 사이드바 기반 작업 공간이 나타난다.

## Non-goals
- 실제 인증 처리, Keycloak 연동, 세션 관리 (목업 버튼만 존재)
- 실제 파일 업로드·인덱싱·진행률 스트리밍 (상태는 정적 목업 데이터)
- 실제 AI 응답 호출·스트리밍·출처 계산 (하드코딩된 예시 대화 1세트)
- 반응형 모바일 레이아웃 정밀 대응 (데스크톱 우선, 모바일은 최소 깨지지 않는 수준)
- 접근성 심화 대응·키보드 단축키·다국어 (한국어 단일)
- 빈 상태/에러/로딩 상태의 모든 변형 (대표 상태 1개씩만)

## Capabilities
- `shared-layout` — 인증 구역의 공통 골격: 워드마크, "새 챗 시작" CTA, 내비게이션(내 문서·챗), 최근 대화 목록, 유저 푸터(로그아웃). 로그인 화면에는 적용되지 않는다.
- `login-page` — 공개 영역 로그인 화면. 이메일/비밀번호 입력과 Keycloak SSO 진입점을 한 장의 카드로 제공. 성공 시 내 문서 화면으로 이동하는 흐름의 진입점.
- `doc-upload-page` — 내 문서 화면. 드롭존과 업로드된 문서 목록, 각 문서의 처리 상태(인덱싱 완료 / 처리 중)를 한눈에 보여준다.
- `chat-page` — 챗 화면. 사용자/AI 메시지 교환 영역, 메시지 작성 영역, AI 답변에 붙는 출처 인용 표기를 제공한다.

## UX Flow
1. 비로그인 사용자는 어느 경로로 진입해도 로그인 화면을 먼저 본다.
2. 로그인 화면에서 로그인 버튼 또는 SSO 버튼을 누르면 내 문서 화면으로 이동한다(인증 성공으로 가정).
3. 내 문서 화면에서 사용자는 문서를 업로드하거나 기존 문서 상태를 확인한다. 사이드바의 "챗" 메뉴 또는 "새 챗 시작" CTA로 챗 화면으로 이동한다.
4. 챗 화면에서 사용자는 메시지를 입력해 AI와 대화하고, AI 답변에 표시된 출처를 통해 어떤 문서에서 근거가 왔는지 확인한다.
5. 사이드바 유저 푸터의 로그아웃을 누르면 로그인 화면으로 돌아간다.
- 예외: 로그인하지 않은 상태로 인증 영역 URL에 접근 시 로그인 화면으로 돌려보낸다(목업 수준의 가드).

## Design Tokens Applied
- 충실도: `guide` — 디자인 시스템의 토큰·톤을 기본으로 삼되, 프로젝트 원칙(중립 팔레트·해요체·테두리 중심)이 충돌 시 우선한다.
- 반영 대상: 중립 팔레트, Geist Sans/Mono, 버튼/카드 라디우스 기준, 그림자 최소화(팝오버만 약한 그림자), 한국어 해요체 문구.
- 원본과 달리 간 결정: 없음(현재 시점). 스펙 단계에서 충돌이 확인되면 이 절에 표로 추가한다.
