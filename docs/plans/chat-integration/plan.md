# 기획서: chat-integration

## Meta
- type: feature
- package: frontend
- design-ref: none

## Problem
챗 화면은 현재 하드코딩된 시드 메시지와 700ms 딜레이의 목업 AI 응답으로 동작하고, 사이드바의 "최근 대화"도 고정값이다. 사용자는 본인의 실제 대화 기록을 저장·재방문할 수 없고, AI 응답이 어떤 문서에 근거했는지 확인할 수 없다. 인증·파일 연동이 완료되어 개인 문서가 실제로 보관되는 상태이므로, 챗 API를 연동해 개인 지식에 근거한 대화가 실제로 동작하는 제품으로 전환해야 한다.

## Users (사용자 / 맥락)
- **누구**: 본인 문서를 업로드해 둔 로그인 사용자.
- **맥락**: 업로드한 자료에 관해 자연어로 질문하고, 답변의 근거를 확인하며 과거 대화를 다시 열람한다.
- **원하는 것**: 응답이 타이핑되듯 즉시 흘러나오고, 출처가 표시되며, 대화가 세션 단위로 저장·재개되는 것.

## Goals
- 사용자가 질문하면 AI 응답이 스트리밍으로 점진적으로 표시된다.
- 응답 하단에 참고한 문서(파일명·페이지)가 출처로 표시된다.
- 대화가 세션 단위로 저장되어 사이드바 "최근 대화"에서 다시 열 수 있고, 필요 시 삭제할 수 있다.
- 새로고침·재방문 시 기존 세션의 전체 메시지가 복원된다.

## Non-goals
- 메시지·세션 편집(제목 수동 변경, 메시지 수정/삭제), 대화 공유·내보내기는 범위 외.
- 응답 중단/재생성, 다중 모델 선택, 프롬프트 템플릿은 범위 외.
- 출처 인용 클릭 시 원본 문서 뷰어 이동 등 확장 UX는 범위 외(파일명·페이지 표시까지만).
- 세션 제목 자동 요약·재생성은 백엔드가 제공하지 않으므로 범위 외(첫 메시지 기반 표시로 갈음).

## Capabilities
- `chat-session` — 새 세션 생성, 내 세션 목록 조회, 단일 세션 + 전체 메시지 조회, 세션 삭제. 사이드바 "최근 대화"를 실제 세션 목록으로 대체하고, 세션 진입 시 메시지 히스토리를 복원한다. 제목이 없는 세션은 첫 사용자 메시지 내용을 기반으로 표시한다.
- `chat-message` — 사용자 메시지를 현재 세션에 전송하고, SSE 스트림으로 받은 토큰을 순서대로 이어붙여 AI 응답 말풍선을 점진적으로 갱신한다. `sources` 이벤트가 도착하면 응답 하단에 출처(파일명·페이지)를 표시하고, `done`에서 스트림을 종료하며, `error`에서 사용자에게 오류를 안내한다.

## UX Flow (사용자 흐름)

### 정상 흐름
1. 로그인 사용자가 `/chat`에 진입하면 최근 대화 목록이 사이드바에 표시된다.
2. 새 대화를 시작하거나 기존 대화를 선택하면 해당 세션의 메시지가 복원된다.
3. 질문을 전송하면 사용자 메시지가 즉시 표시되고, AI 응답이 스트리밍으로 점진 표시된다.
4. 응답이 끝나면 참고한 문서(파일명·페이지)가 응답 아래 출처로 표시된다.
5. 사이드바에서 과거 대화를 다시 열거나 불필요한 대화를 삭제할 수 있다.

### 예외 흐름
- 스트리밍 중 네트워크 단절·`error` 이벤트 수신 시, 진행 중이던 응답 상태를 정리하고 사용자가 이해할 수 있는 오류 메시지를 안내한다.
- 세션 조회·삭제·생성 API 실패 시 목록/화면 상태가 깨지지 않도록 재시도 또는 안내한다.
- 참고 문서가 없는 응답은 출처 영역을 생략한다.
- 미로그인 상태로 진입 시 기존 session-guard 규칙에 따라 로그인 화면으로 유도된다.

## Constraints
- 메시지 전송 엔드포인트는 SSE(`text/event-stream`)이며 인증 헤더가 필요해 `EventSource`를 사용할 수 없다 → `fetch + ReadableStream` 방식으로 수신해야 한다.
- BFF(`/api/[...proxy]/`)가 SSE 응답을 버퍼링 없이 pass-through하도록 스트리밍 응답 그대로 중계해야 한다. `Authorization`은 BFF가 쿠키에서 주입한다.
- 세션 제목은 백엔드가 자동 생성하지 않으며 초기값이 null이다 → 프론트엔드가 첫 메시지 내용으로 표시 규칙을 가진다.
- SSE 이벤트 타입은 `token` / `sources` / `done` / `error` 네 종류로 고정이며 수신측이 모두 처리해야 한다.

- 스트리밍 중 사용자가 세션을 이탈/전환할 때 진행 중 스트림 취소 정책 — spec 단계에서 결정.
- 세션 제목 표시 규칙(첫 메시지 앞 N자 등 구체값) — spec 단계에서 결정.
- 세션 삭제 확인 UX 및 목록 갱신 방식(낙관적 갱신 여부) — spec 단계에서 결정.
- 출처 표시 형식의 세부(중복 파일·페이지 병합 여부 등) — spec 단계에서 결정.
- 스트림 오류·중단 시 부분 응답 보존 여부 — spec 단계에서 결정.
